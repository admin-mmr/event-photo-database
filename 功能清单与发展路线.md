# 湘舍动公益文件系统 — 功能清单与发展路线

**系统名称**：湘舍动公益文件系统 / MM Runners Photo Archive  
**当前版本**：1.0（Google Apps Script）  
**更新日期**：2026 年 4 月  
**当前状态**：五个开发阶段全部完成 ✅

---

## 第一部分 — 已实现功能（v1.0）

### 1.1 身份认证与权限管理

| 功能 | 说明 |
|------|------|
| Google OAuth 单点登录 | 用户使用现有 Google 账号登录，无需单独注册或管理密码 |
| 基于角色的访问控制 | 三种角色：`admin`（管理员，完整权限）、`user`（普通用户，上传与浏览）、`api_client`（仅限 REST API 程序化访问） |
| 基于会话的鉴权 | 每次请求均通过 GAS 会话验证当前登录账号，无需客户端管理令牌 |
| 停用账号拦截 | 已停用的用户在登录时被拒绝访问，并显示清晰的错误提示及联系管理员的指引 |
| 管理员自助操作界面 | 所有用户与俱乐部管理均通过 Web 界面完成，初始部署后无需手动编辑电子表格 |

### 1.2 用户管理（管理员）

| 功能 | 说明 |
|------|------|
| 添加用户 | 管理员填写邮箱、所属俱乐部和角色，记录立即写入 Users 工作表 |
| 邮箱规范化 | 用户邮箱以小写形式存储和匹配，不区分大小写 |
| 编辑用户 | 管理员可随时更新用户的所属俱乐部或角色 |
| 停用用户 | 停用账号后用户无法登录，但历史上传记录完整保留；停用用户在列表中显示为灰色 |
| 重新激活用户 | 可一键重新激活已停用的用户账号 |
| 分页用户列表 | 用户列表分页展示，确保大量用户时页面响应流畅 |
| 批量导入用户 | 对于大批量导入，可直接按列顺序粘贴行数据到 Users 工作表 |

### 1.3 俱乐部管理（管理员）

| 功能 | 说明 |
|------|------|
| Clubs 工作表 | 管理员维护的 `Clubs` 工作表是已批准跑步俱乐部的唯一数据来源 |
| 自动初始化 | 首次运行时，系统自动从内置默认列表（New Bee、Misty Mountain、Nankai、Admin）初始化 Clubs 工作表 |
| 添加俱乐部 | 管理员可通过 Admin UI 添加新俱乐部，需填写展示名称和规范化文件夹名称 |
| 编辑俱乐部 | 管理员可更新俱乐部的展示名称或规范化名称 |
| 停用 / 重新激活俱乐部 | 已停用的俱乐部不再出现在上传和注册选择器中，历史数据完整保留 |
| 规范化名称校验 | 规范化名称须符合格式 `[A-Za-z][A-Za-z0-9]*(_[A-Za-z][A-Za-z0-9]*)*`，确保可安全用作 Drive 文件夹名 |

### 1.4 活动管理（管理员）

| 功能 | 说明 |
|------|------|
| 创建活动 | 管理员输入活动名称和日期；系统自动生成文件夹名（`YYYY-MM-DD_活动名称`）并在 Drive 中创建对应文件夹 |
| 文件夹名称实时预览 | 管理员输入信息时，页面实时显示将要在 Drive 中创建的文件夹名称 |
| 重复活动拦截 | 若相同文件夹名已存在，系统阻止创建并显示清晰的错误提示 |
| 编辑活动 | 管理员可更新活动的展示名称或日期（Drive 文件夹名创建后不可修改） |
| 可排序活动列表 | 活动列表默认按日期从新到旧排列；点击日期列标题可切换升序/降序 |
| 日期范围过滤 | 管理员可通过日期范围过滤活动列表 |
| 直达 Drive 链接 | 每个活动行均有文件夹图标，点击在新标签页中打开对应 Google Drive 文件夹 |
| 不允许删除 | 活动永不删除，以防止已关联上传记录的数据丢失 |

### 1.5 照片上传（所有用户）

| 功能 | 说明 |
|------|------|
| 四步引导式上传流程 | 第 1 步：选择活动 → 第 2 步：查看俱乐部现有内容 → 第 3 步：选择文件 → 第 4 步：上传结果摘要 |
| 活动卡片网格（含日期过滤） | 用户以卡片网格形式浏览活动，日期范围过滤器帮助快速定位目标活动 |
| 俱乐部文件夹查看（只读） | 上传前，用户可查看本俱乐部在该活动下已有的上传记录 |
| 多文件选择 | 浏览器文件选择器支持一次选取多张照片 |
| 文件类型过滤 | 仅接受 JPEG、PNG 和 HEIC 格式；其他类型（PDF、视频、RAW 等）自动跳过并在摘要中统计 |
| 文件大小限制 | 单张照片不超过 50 MB；单次上传批次总大小不超过 200 MB；超出限制的文件在上传前标记错误 |
| 重复文件检测 | 每个文件与俱乐部在该活动下的所有现有文件按文件名和字节大小进行比对，精确匹配标记为重复 |
| 重复文件处理选择 | 检测到重复时，用户可逐一选择跳过或覆盖 |
| 自动创建批次文件夹 | 每次上传会话自动创建一个第三层文件夹：`YYYYMMDD-HHMMSS_用户名` |
| 保留原始文件名 | 文件以原始文件名平铺存储到批次文件夹，不做任何改名 |
| 逐文件上传进度 | 上传进度逐文件展示；上传过程中按钮禁用，防止重复提交 |
| 上传结果摘要 | 上传完成后显示：已上传照片数、总大小、跳过重复数、跳过非照片文件数 |

### 1.6 上传日志

| 功能 | 说明 |
|------|------|
| Upload_Log 工作表 | 每次上传会话写入一条记录：活动 ID、俱乐部、上传者、批次文件夹名/ID、文件数、总大小、跳过数、时间戳、来源 |
| 来源标记 | 记录上传来自 Web 应用（`web_app`）还是 REST API（`api`） |
| 只追加不修改 | 上传日志仅追加写入，不可编辑或删除 |
| 管理员可见 | 所有上传日志可在汇总仪表盘及 Google 工作表中直接查看 |

### 1.7 管理员汇总与核查仪表盘

| 功能 | 说明 |
|------|------|
| 日期范围选择器 | 管理员选定日期范围后，所有汇总数据自动筛选 |
| 按活动统计上传情况 | 对范围内每个活动：列出已上传俱乐部、照片数量、总大小 |
| 无上传活动列表 | 单独展示"待关注"列表，高亮显示零上传记录的活动 |
| 文件夹命名违规检测 | 扫描 Drive 第一层和第二层文件夹，查找命名规范违规，列出文件夹名称及违规类型 |
| CSV 导出 | 完整汇总报告可导出为 CSV 文件 |
| 异常通知邮件 | 检测到违规时，管理员可一键发送通知邮件 |

### 1.8 文件夹命名规范执行

| 功能 | 说明 |
|------|------|
| 第一层校验 | 活动文件夹须符合格式 `YYYY-MM-DD_标题词_标题词`；日期部分须为有效日历日期 |
| 第二层校验 | 俱乐部文件夹须与 Clubs 工作表中的已批准俱乐部名称一致 |
| 第三层自动生成 | 上传批次文件夹始终由系统自动生成，用户不可自定义命名 |
| 违规扫描 | `scanLayer1Violations()`、`scanLayer2Violations()`、`scanAllViolations()` — 可从汇总仪表盘和活动管理页按需调用 |
| 违规警告横幅 | 管理员活动页后台检测到违规时，页面顶部显示橙色警告横幅 |

### 1.9 跨组织 REST API

| 功能 | 说明 |
|------|------|
| `api_check_folder`（GET） | 根据活动文件夹名返回 Drive 文件夹 ID，不存在则返回 `null` |
| `api_list_files`（GET） | 根据 Drive 文件夹 ID 返回文件 JSON 列表（文件名、大小、MIME 类型、修改时间） |
| `api_upload_file`（POST） | 接收 Base64 编码照片 + 元数据（活动 ID、俱乐部名）；执行完整的校验与重复检测流程；写入 Upload_Log |
| API 密钥鉴权 | API 密钥通过 `?api_key=` 查询参数传递（GAS Web App 无法读取 HTTP 请求头）；密钥即 api_client 用户的邮箱地址 |
| 请求频率限制 | 每个 API 密钥每小时限 60 次请求；计数器通过 `Rate_Limit` 工作表的固定时间窗口方式记录 |
| 标准化错误响应 | 所有 API 响应遵循格式 `{ status, code, message, data }`，使用标准 HTTP 风格状态码（200、400、403、409、415、429、500） |
| 合作伙伴客户端示例 | 提供可直接复制使用的 GAS 客户端脚本（`example/partner-client.gs`） |

## 第二部分 — 未来功能展望（v2 及更高版本）

以下功能因 Google Apps Script 的技术限制、开发范围或实现复杂度未纳入 v1。这些功能是计划中的 v2（Node.js + Firebase）迁移的重点候选项。

### 2.1 高优先级

| 功能 | 价值 | 备注 |
|------|------|------|
| **基于 EXIF 的重复检测** | 即使文件名不同（如手机相册重命名），也能识别真正的重复照片 | 需要读取图片二进制数据；GAS 50 MB 载荷限制在 v1 中不可行 |
| **RAW 格式照片支持** | 允许专业摄影师上传 CR2、ARW、NEF 等格式 | 存储与预览成本较高；建议设置为管理员可按活动开启的选项 |
| **文件浏览器缩略图预览** | 用户可直观确认上传内容；管理员可快速核查文件夹 | GAS HtmlService 可提供 Drive 缩略图 URL；v1 技术可行但速度慢 |
| **管理员操作审计日志** | 追踪所有管理员操作（谁在何时添加/编辑/停用了谁） | 追加写入审计工作表；v1 中可直接添加 |
| **定时文件夹违规扫描** | 主动发现命名违规，而非仅在手动请求时检测 | GAS 时间触发器可每日运行；需增加专用违规记录工作表 |

### 2.2 中等优先级

| 功能 | 价值 | 备注 |
|------|------|------|
| **管理员批量下载** | 管理员可将指定活动/俱乐部的所有照片打包下载为 ZIP | 使用 Google Drive 文件夹下载 API；可从汇总仪表盘触发 |
| **照片搜索与标签** | 按文件名搜索；管理员可添加活动级标签供所有用户查看 | 需要标签工作表或 Firestore；v2 的理想功能 |
| **活动归档 / 软删除** | 允许管理员将旧活动标记为已归档，从活动视图中隐藏但不丢失数据 | 在 Events 工作表添加 `status` 列；在 UI 中过滤归档活动 |
| **多管理员通知** | 异常邮件发送至可配置的管理员邮箱列表，而非仅发给当前操作管理员 | 在脚本属性中添加 `notification_emails` 配置项 |
| **大批量上传实时进度** | 展示整批次的实时进度（如"12 张中已上传 3 张"） | 当前实现逐文件更新；整体进度条为 UI 增强 |
| **活动重新开放** | 允许管理员在活动结束后重新开放额外上传 | UI 开关；无需修改 Drive 结构 |

### 2.3 长期规划（v2 迁移）

| 功能 | 价值 | 备注 |
|------|------|------|
| **迁移至 Node.js + Firebase** | 突破 GAS 执行时间限制，支持后台任务、Webhooks 和更高性能查询 | 命名规范、文件夹结构和所有业务逻辑保持不变 |
| **Firestore 数据库** | 替换 Google Sheets 作为数据库，提供更快的查询速度和实时更新能力 | 迁移路径：导出 Sheets CSV → 导入 Firestore |
| **Webhook 通知** | 上传完成、检测到违规或触达配额限制时推送通知（Slack、邮件等） | GAS 不支持出站 Webhooks；需要 Node.js + Cloud Functions |
| **外部 Drive 文件夹导入** | 允许合作伙伴共享 Drive 文件夹，系统自动导入其目录结构 | 复杂的树遍历；适合 Node.js 实现 |
| **视频支持** | 扩展上传流程以接受 MP4/MOV 格式 | 存储和处理成本需单独设计；GAS 50 MB 载荷限制是 v1 阻碍 |
| **移动端优化** | 针对手机摄影上传优化的轻量 UI 或 PWA | 当前 UI 已支持响应式布局，但未针对触屏优化；v2 中完善 |
| **多组织支持** | 在同一系统内隔离管理多个跑步组织的数据 | 需要在 Drive 和 Sheets 层面实现租户隔离；v2 的架构变更 |
| **照片数据分析仪表盘** | 按俱乐部、活动和时间段展示上传趋势图表 | Sheets 中的聚合查询性能有限；Firestore + BigQuery 是正确的技术栈 |
| **API 客户端服务账号 OAuth** | 将"邮箱作为 API 密钥"替换为标准 OAuth 2.0 服务账号 | 安全性更高；需要合作伙伴组织创建 GCP 服务账号 |

## 总结

**v1.0 功能完整。** 五个规划阶段全部实现并经过全面测试（所有模块代码覆盖率 ≥ 85%）。系统已达到生产就绪状态，完全满足当前湘舍动公益组织的运营规模。

下一阶段的主要投入是 **v2.0 迁移**（Node.js + Firebase），在保留所有现有文件夹命名规范和用户操作流程的基础上，实现后台处理、Webhooks 和规模化性能提升。
